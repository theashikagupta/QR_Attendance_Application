# QR Based College Attendance Management System

A **full-stack, role-based attendance platform** built with **React + Node.js/Express + MongoDB + Python/Flask** to digitize classroom attendance using QR codes.

This project is designed around a practical college workflow:
- **Admin** manages students, teachers, section allocations, reports, and issue resolution
- **Teacher** logs in, views assigned sections, scans QR codes, marks attendance, and contacts admin when needed
- **QR service** validates QR payloads and is structured to support future secure encryption/report generation workflows

> **Why this project matters:** It is not a toy CRUD app. It combines authentication, role-based access control, scheduling rules, attendance validation, QR workflows, reporting, and cross-service integration in one system.

## Architecture Overview

This repository is a **monorepo** with 3 separate applications:

```text
QR_Attendance_Application/
├── frontend/      # React client
├── backend/       # Express + MongoDB REST API
└── qr-service/    # Flask QR validation/report service
```

### How the services interact

1. **Frontend** authenticates Admin/Teacher and calls backend APIs.
2. **Backend** manages users, students, teachers, section allocations, and attendance records.
3. **QR service** validates QR payloads when configured through `QR_SERVICE_BASE_URL`.
4. **Backend** applies business rules such as:
   - teacher-allocation ownership
   - section/department visibility match
   - time-slot enforcement
   - duplicate attendance prevention
   - late marking logic

## Core Features

## 1) Authentication & Role-Based Access
- JWT-based login for **Admin** and **Teacher**
- Protected routes on backend for role-specific access
- Default admin bootstrap using environment variables
- Token stored in frontend local storage and attached automatically through Axios interceptors

## 2) Admin Module
- Create students
- Update student profile/status
- Delete students
- Search/filter students by:
  - department
  - section
  - semester
  - status
- Create teachers with autogenerated temporary password
- Email teacher credentials via SMTP
- Allocate sections to teachers with subject and time slot
- View students under a section allocation
- Manual attendance override with reason
- Review teacher-raised contact requests
- Resolve contact requests with response message
- View reports:
  - attendance summary
  - defaulters list

## 3) Student QR Workflow
- Student record creation generates:
  - unique `qrSecret`
  - QR payload
  - QR image as Base64 data URL
- Current payload format used in backend utility:
  - `ENC::<studentId>::<qrSecret>`
- Student status controls attendance eligibility:
  - `ACTIVE`
  - `INACTIVE`
  - `DEBARRED`

## 4) Teacher Module
- View own teacher profile and allocations
- View students in assigned section only
- Scan QR and mark attendance
- Auto-checks before marking attendance:
  - allocation belongs to logged-in teacher
  - scan happens inside assigned slot
  - QR payload is valid
  - student exists
  - student belongs to correct department/section
  - student is not inactive/debarred
  - attendance is not already marked for the day
- Late marking logic:
  - attendance is marked **LATE** if scanned more than **10 minutes after slot start**
- Generate **session QR** for an allocation
- Submit issue/contact requests to admin

## 5) Reports & Analytics
- Attendance summary with filters
- Defaulters list using configurable threshold (default 75%)
- Teacher issue/request tracking

## 6) QR Service
The Flask service currently includes:
- QR payload generation endpoint
- QR payload validation endpoint
- health check endpoint

It also contains **stub endpoints** for:
- PDF report export
- Excel report export

That means the service is structured for expansion, but **PDF/Excel generation is not fully implemented yet**.

## 2. Setup Backend

```bash
cd backend
npm install
```

### Create `backend/.env`

```env
PORT=9000
MONGODB_URI=your_mongodb_connection_string
CLIENT_ORIGIN=http://localhost:5173

JWT_SECRET=your_jwt_secret
JWT_EXPIRES_IN=8h

DEFAULT_ADMIN_EMAIL=admin@example.com
DEFAULT_ADMIN_PASSWORD=admin12345

SMTP_HOST=smtp.example.com
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=your_email@example.com
SMTP_PASSWORD=your_email_password
SMTP_FROM_NAME=QR Attendance Admin
SMTP_FROM_EMAIL=your_email@example.com

QR_SERVICE_BASE_URL=http://localhost:7000
```

### Run backend

```bash
npm run dev
```

Backend runs on:
- `http://localhost:9000`

Health check:
- `GET http://localhost:9000/health`

## 3. Setup QR Service

Move back to project root first:

```bash
cd ../qr-service
```

### Create virtual environment

**Windows (PowerShell):**
```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
```

**macOS / Linux:**
```bash
python3 -m venv .venv
source .venv/bin/activate
```

### Install dependencies

```bash
pip install -r requirements.txt
```

### Create `qr-service/.env`

```env

FLASK_RUN_PORT=7000
PORT=7000
SECRET_KEY=change-this-flask-secret
CORS_ORIGINS=http://localhost:5173
BACKEND_BASE_URL=http://localhost:9000
```

### Run QR service

```bash
python main.py
```

QR service runs on:
- `http://localhost:7000`

Health check:
- `GET http://localhost:7000/health`

## Known Limitations

Being honest here is better than pretending everything is complete:

- QR payload format currently uses a simple placeholder structure in parts of the flow: `ENC::<studentId>::<qrSecret>`
- Flask QR service includes TODOs for stronger cryptographic validation
- PDF and Excel report export endpoints exist but currently return **501 Not Implemented**
- There is no student-facing portal yet
- Token persistence is local-storage based, so production hardening can be improved further with secure cookie/session strategies

These are not weaknesses if you can explain them clearly in interviews. They are opportunities to discuss how you would evolve the system.
